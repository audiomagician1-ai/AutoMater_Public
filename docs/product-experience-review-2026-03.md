# 智械母机 AutoMater — 产品体验全面复盘报告

> **日期**: 2026-03-02  
> **视角**: 产品经理  
> **审查范围**: 13 个前端页面 + 10 个组件 + 43 个后端引擎模块 + 全部项目文档  
> **项目版本**: v6.0+（含暂存区 v7.0~v12.0 功能）

---

## 〇、执行摘要

AutoMater 在功能完整度上已经非常出色——5 阶段流水线、42+ 工具体系、多工作流预设、元 Agent 管家、实时监控仪表盘、团队热加入、会话追踪等核心能力一应俱全。**但作为一款面向「非程序员或需要快速原型的开发者」的产品，当前版本在体验层面存在大量摩擦点，严重制约了从"功能 Demo"到"可用产品"的跨越。**

本报告从产品经理视角出发，系统性梳理 **7 大类共 42 个体验问题**，按严重程度分为 P0（阻断）/ P1（严重影响）/ P2（体验损伤）/ P3（打磨优化）四级。

### 核心结论

| 维度 | 得分 | 关键瓶颈 |
|------|------|---------|
| 首次上手 (FTUE) | ★★☆☆☆ | 新用户 5 分钟内无法完成第一次许愿 → 流失 |
| 核心流程可靠性 | ★★☆☆☆ | 全链路从未经过真实运行时验证（仅 tsc） |
| 信息架构 | ★★★☆☆ | 13 个页面缺乏层级，侧边栏导航过载 |
| 可观测性 | ★★★☆☆ | 日志多但缺乏结构化，出问题用户无法自救 |
| 错误处理 | ★★☆☆☆ | 错误消息对非程序员几乎不可读 |
| 视觉一致性 | ★★★★☆ | 科技感统一，但信息密度偏高 |
| Token/成本控制 | ★★★☆☆ | 有架构但缺用户侧可视化和预算硬限制 |

## 一、首次使用体验 (FTUE) — 5 个问题

### P0-01: 无引导式 Onboarding 流程
**现象**: 新用户打开应用，直接看到一个空白的项目列表页 + 16px 的"还没有项目"提示。没有欢迎弹窗、没有引导动画、没有步骤指引。  
**影响**: 目标用户是"非程序员"，他们不知道第一步该做什么。"设置 LLM API Key"这个前置条件隐藏在底部一行小字的 amber 提示条中，极易被忽略。  
**建议**: 
- 首次启动弹出全屏 Onboarding 向导（3 步：选 Provider → 填 Key → 测试连接 → 创建第一个项目）
- 侧边栏底部的绿/黄状态灯加上 tooltip 解释

### P0-02: LLM 配置门槛过高
**现象**: SettingsPage 直接要求用户填入 Provider URL、API Key、模型名称。对于非程序员用户，"什么是 API Key"、"去哪里获取"、"OpenAI Compatible 是什么意思"都是巨大认知障碍。  
**影响**: 这是使用产品的硬性前提，如果这一步卡住，用户将直接放弃。  
**建议**: 
- 提供主流 Provider（OpenAI / Anthropic / OpenRouter）的预设模板，一键选择后只需填 Key
- 在输入框旁添加"获取 Key 教程"外链按钮
- 添加"一键测试连接"并明确显示"连接成功/失败"状态

### P1-03: 项目创建流程信息量过大
**现象**: 创建表单一次性展示了项目名、工作区路径、历史版本路径、Git 模式（本地/GitHub）、GitHub Token 等大量字段。  
**影响**: 非程序员根本不理解"工作区路径"、"历史版本文件夹"这些概念。理想的新手体验应该是只输入一个项目名就能开始。  
**建议**:
- MVP 模式：只需输入项目名，其他全部自动默认
- 高级选项折叠隐藏，给"知道自己在做什么"的用户用
- 工作区路径默认为 `~/AutoMater-Projects/{项目名}` 且创建时自动 mkdir

### P1-04: "许愿"后用户失去控制感
**现象**: 用户在 WishPage 提交需求后，代码直接调用 `project.start()` 跳转到 OverviewPage。用户没有任何确认环节——"你确认要用这段描述启动 AI 开发吗？"  
**影响**: 一旦启动就开始消耗 Token/API 费用。非程序员可能只是在试探性地输入，不理解"提交就等于花钱"。  
**建议**:
- 提交前显示确认弹窗："即将启动 AI 团队开发，预计消耗约 X Token（约 $Y）。确认启动？"
- 或在 OverviewPage 上先停留在"就绪"状态，等用户主动点击"启动"

### P2-05: GuidePage 教程虽全但入口深
**现象**: 教程页面内容质量不错（8 篇，覆盖从入门到进阶），但入口在侧边栏的第 12 个 Tab 位置。新用户几乎不会翻到那里。  
**影响**: 精心编写的教程内容无法触达最需要它的新用户。  
**建议**: 首次启动或空项目列表时，直接在主区域嵌入教程卡片入口

## 二、核心流程可靠性 — 6 个问题

### P0-06: 全部功能仅通过 tsc 编译验证，从未经过运行时验证
**现象**: 根据暂存区记录，v7.0~v12.0 的所有功能（元 Agent 管理、对话备份、热加入、实时监控、工作流预设、品牌更名）都只进行了 `tsc --noEmit` 编译检查，没有实际启动应用运行过。  
**影响**: 这意味着当前版本很可能存在大量运行时错误——IPC 通道不匹配、DB schema 缺列、模块初始化顺序错误、前端组件渲染异常等。任何用户拿到这个版本，第一次运行很可能就会崩溃。  
**建议**: 
- 立即制定一个最小运行时验证计划：启动应用 → 创建项目 → 提交需求 → 观察流水线至少启动 Phase 1
- 建立 CI 级别的冒烟测试（至少 electron-builder 成功 + 主窗口能打开）

### P0-07: 品牌更名后残留的 "F" Logo
**现象**: Sidebar.tsx 第 42 行，品牌 Logo 仍然显示字母 "F"（来自 AgentForge 首字母）。项目已更名为"智械母机 AutoMater"，但 Logo 文字未更新。  
**影响**: 品牌一致性破坏。用户看到 "F" 不知道代表什么。  
**建议**: 更新为 "A" 或使用 AutoMater 的图标

### P1-08: 并发防护虽已实现但 UI 反馈缺失
**现象**: 后端 `agent-manager.ts` 有并发防护（`registerOrchestrator` 检查），前端 OverviewPage 有 1.5 秒防抖。但如果用户快速双击"启动"被拦截，UI 没有任何反馈——按钮不变灰、没有 toast、没有日志。  
**影响**: 用户以为操作没生效，继续点击，产生困惑。  
**建议**: 被拦截时弹出轻量 toast "项目已在运行中"

### P1-09: 熔断器状态对用户不可见
**现象**: 后端有完整的 circuit-breaker 机制（NonRetryableError 分类、DB 记录 last_failure_reason），但前端没有任何地方展示"为什么这个 Feature 反复失败"、"模型是否不可用"。  
**影响**: 用户看到 Feature 标记为 failed，点击"继续"，再次失败，不知道是自己的问题还是系统的问题。  
**建议**: 
- OverviewPage 或 BoardPage 的 failed Feature 卡片上显示失败原因摘要
- 如果是模型不可用等系统级问题，在顶部显示全局告警条

### P1-10: 续跑（Resume）缺乏分诊结果展示
**现象**: 后端 `detectImplicitChanges()` 会分诊用户的新需求（全新 vs 迭代变更），但分诊结果只写入日志，不在 UI 上结构化展示。用户不知道"续跑到底会做什么"。  
**影响**: 用户对续跑行为没有预期，产生不确定感。  
**建议**: 续跑前在 OverviewPage 弹出分诊摘要卡片："检测到 3 个新需求 + 2 个迭代变更。确认继续？"

### P2-11: 停止/暂停无确认弹窗
**现象**: 点击"暂停"直接停止所有 Agent 工作，没有二次确认。可能正在进行的 LLM 调用被强制中断。  
**影响**: 误触暂停会导致进行中的工作丢失（虽然有 AbortController 和 checkpoint，但用户不知道）。  
**建议**: 添加确认弹窗："当前有 N 个 Agent 正在工作，暂停后进度会保存。确认暂停？"

## 三、信息架构与导航 — 7 个问题

### P1-12: 侧边栏 12 个 Tab 导航过载
**现象**: 项目内侧边栏有 12 个导航项：全景、许愿、看板、文档、工作流、团队、上下文、时间线、会话、产出、日志、教程。每个 Tab 是 12×12 的小按钮 + 10px 文字。  
**影响**: 
- 对非程序员用户来说，"上下文"、"时间线"、"会话" 这些概念完全超出认知
- 12 个等权重并列的入口让用户不知道该看哪个
- 在 1080p 屏幕上 12 个按钮几乎占满侧边栏垂直空间，没有视觉分组

**建议**: 
- 分组分层：核心（全景 / 看板 / 产出）→ 管理（许愿 / 团队 / 工作流）→ 诊断（日志 / 时间线 / 上下文 / 会话）
- 诊断类默认折叠，高级用户展开
- 或采用顶部 Tab 替代部分深层功能

### P1-13: "全景"页信息密度失控
**现象**: OverviewPage 是最长的页面（1459 行代码），从上到下依次堆叠了：
1. 指挥中心 Header + 状态控制条
2. 实时监控（CPU/GPU/内存 4 面板）
3. 活动趋势图（Token/成本/代码/API 4 折线图）
4. 导入进度卡片（条件显示）
5. 进度概览（环形图 + 5 个统计卡 + 4 个汇总卡）
6. 流水线进度条 + 文档完成度
7. 验收提示
8. 三层架构图（SVG DAG，支持缩放/拖拽/下钻）
9. Feature 路线图列表
10. Canvas 粒子背景

**影响**: 用户打开这个页面会感到信息过载。核心的"项目现在怎么样了"被淹没在各种图表中。  
**建议**: 
- 将"实时监控"和"活动趋势"移到独立的"监控"Tab，它们不是每次打开都需要看的
- 全景页只保留：状态控制条 + 进度概览 + 流水线 + 架构图
- 使用可折叠 section 或 Tab 化布局

### P2-14: 许愿页布局不对称
**现象**: WishPage 左侧是 272px 固定宽度的需求列表，右侧是元 Agent 对话区。但如果没有需求，左侧显示一个"暂无需求"的空状态，右侧是完整的对话界面。  
**影响**: 新用户第一次进来，左侧空白 + 右侧一个 AI 聊天框，不确定应该在左侧"+ 需求"还是直接在右侧对话。两个入口完成同一件事（提需求），但行为不同（左侧提交 = 直接启动开发；右侧对话 = 元 Agent 分诊后可能创建需求）。  
**建议**: 
- 新项目首次进入时，隐藏左侧列表，全屏显示一个引导式的需求输入区
- 明确区分"正式提交需求并启动"和"和管家聊聊"两个动作

### P2-15: 看板页与全景页 Feature 信息重复
**现象**: OverviewPage 底部的"Feature 路线图"和 BoardPage 展示的是同一组 Feature 数据，只是布局不同（列表 vs 看板列）。  
**影响**: 用户不确定该在哪里管理 Feature。  
**建议**: OverviewPage 只展示汇总摘要（数字 + 进度条），详情引导用户去 BoardPage

### P2-16: 文档页、产出页、日志页功能定位模糊
**现象**: 
- DocsPage：浏览 Agent 自动生成的设计/需求/测试文档
- OutputPage：浏览 Agent 生成的代码文件
- LogsPage：查看 Agent 实时日志

对于用户来说，这三个页面都是"看 Agent 产出的东西"，分成三个 Tab 导致频繁切换。  
**建议**: 考虑合并为一个"产出"页，用顶部 Tab 切换（代码 / 文档 / 日志）

### P2-17: 上下文页 (ContextPage) 对非程序员无意义
**现象**: ContextPage 展示的是 Agent 的上下文窗口使用情况——输入 token 分布、压缩状态等。  
**影响**: 这是面向开发者/高级用户的调试信息，对"非程序员"目标用户完全无意义，反而增加认知负担。  
**建议**: 默认隐藏，仅在"高级/开发者模式"下显示

### P3-18: 时间线页 (TimelinePage) 功能强大但使用场景不明
**现象**: TimelinePage 有 4 个 Tab（时间线/分析/检查点/知识库），是一个完整的项目历史审计工具。  
**影响**: 功能虽好，但在当前阶段（产品尚未跑通一个完整流程），优先级不高。  
**建议**: 可保留，但在导航中降低优先级（归入"高级"分组）

## 四、错误处理与用户反馈 — 6 个问题

### P0-19: 错误消息面向开发者而非用户
**现象**: 各处 catch 块的错误消息直接透传后端异常——如 `❌ Error: Request failed with status code 401`、`NonRetryableError: Model does not exist`。  
**影响**: 非程序员看到这些消息完全不知道该怎么办。  
**建议**: 
- 建立错误码 → 用户友好消息的映射层
- 401 → "API 密钥无效，请检查设置中的密钥是否正确"
- Model not found → "模型名称有误，请在设置中确认模型名称"
- 每个错误消息都附带"该怎么办"的行动指引

### P1-20: 项目删除无二次确认
**现象**: ProjectsPage 的删除按钮 `handleDelete` 直接调用 `project.delete(id)`，没有确认弹窗。  
**影响**: 误删项目将丢失所有数据（Feature、文档、日志、产出）。  
**建议**: 必须添加确认弹窗，且输入项目名才能确认删除（类似 GitHub 删库的交互）

### P1-21: 导入项目失败后无恢复路径
**现象**: `handleImportProject` 的 catch 只写一条日志。如果导入分析中途失败（LLM 超时、模型不可用），项目会处于 `analyzing` 状态，用户不知道该怎么重试。  
**影响**: 项目卡在不可操作的中间状态。  
**建议**: 
- 失败时显示明确的错误卡片 + "重试分析"按钮
- 允许用户跳过分析直接手动设置需求

### P1-22: 轮询刷新策略过于激进
**现象**: 多个页面使用 `setInterval` 轮询刷新：
- ProjectsPage: 6 秒轮询 `loadProjects()`，每个项目还要 `getStats()`
- OverviewPage: 5 秒轮询 `load()`（获取 features + stats + project）
- BoardPage: 4 秒轮询
- WishPage: 5 秒轮询
- App.tsx: 5 秒轮询统计

**影响**: 
- 5 个轮询同时运行时，主进程每秒处理大量 IPC 请求
- 在项目未运行（paused/delivered）时仍在无意义地轮询
- 可能导致 UI 卡顿（Map/Object 频繁重建触发 React re-render）

**建议**: 
- 项目未运行时降低轮询频率或停止轮询
- 用后端事件推送替代前端轮询（已有 IPC 事件基础设施，应充分利用）
- 使用 `React.memo` 和选择性订阅减少 re-render

### P2-23: Toast/Notification 系统缺失
**现象**: 目前所有操作反馈通过两种方式：1) 底部 StatusBar 的最后一条日志滚动显示 2) Electron 原生通知（仅验收阶段）。没有 Toast/Snackbar 组件。  
**影响**: 创建项目成功、导出完成、设置保存等操作完成后，用户感知不到"操作成功了"。  
**建议**: 引入一个轻量 Toast 组件，用于所有关键操作的成功/失败反馈

### P2-24: StatusBar 信息利用率低
**现象**: StatusBar 只有 27px 高度，同时显示 Features、Agents、Tokens、Cost 和最后一条日志。日志被 truncate 到几乎看不到。  
**影响**: 本应是最重要的状态概览区，但因为空间不足信息被压缩到无法阅读。  
**建议**: 
- StatusBar 可点击展开为更大的底部面板
- 或将实时状态摘要移到 OverviewPage 顶部卡片

## 五、可观测性与透明度 — 6 个问题

### P1-25: Agent 工作过程不够透明
**现象**: TeamPage 的 AgentWorkFeed 设计不错（对话式展示思考/工具/结果），但只有在选中某个 Agent 时才能看到。用户没有一个"全局实时动态"的视图来了解"AI 团队现在在做什么"。  
**影响**: 用户提交需求后，除了日志页的大量文字滚动，没有直观的"AI 正在认真工作"的感知。  
**建议**: 
- OverviewPage 添加一个"实时动态"迷你面板：显示最近 5 条 Agent 动作（"PM 正在分析需求"、"Developer-1 正在编写 login.tsx"）
- 或在架构图上直接用动画展示当前 Agent 在哪个模块上工作（已有基础，但不够明显）

### P1-26: Token 消耗无预估、无预算硬限制
**现象**: 用户在提交需求前无法预估这次开发会花多少 Token/钱。运行中虽然 StatusBar 显示累计 cost，但没有"超过预算自动停止"的功能。  
**影响**: 
- 非程序员可能不理解"一次许愿可能花费 $5-50"
- 没有预算硬限制可能导致意外高额费用

**建议**: 
- 创建项目时增加"预算上限"字段（默认 $10），到达时自动暂停
- 提交需求时估算费用范围（基于 wish 长度和历史数据）

### P1-27: 日志页海量文本缺乏结构
**现象**: LogsPage 以纯文本形式展示所有日志。一个完整开发周期可能产生数千条日志。虽然有按 Agent 过滤和关键词搜索，但缺乏按阶段分组、按严重程度过滤等结构化视图。  
**影响**: 当出现问题时，用户很难在海量日志中定位关键信息。  
**建议**: 
- 增加日志级别过滤（info / warning / error）
- 按 Phase 分组折叠
- 错误日志高亮 + 快速跳转

### P2-28: 文档生成过程不可见
**现象**: 后端 doc-manager 会在各阶段生成设计文档、需求文档、测试规格。但在文档生成过程中，DocsPage 只是静态展示已有文档，没有"正在生成中"的实时反馈。  
**影响**: 用户不知道文档什么时候会出现，需要反复刷新查看。  
**建议**: 添加"生成中"骨架屏 + 实时进度指示

### P2-29: 产出页缺乏代码质量指标
**现象**: OutputPage 展示生成的文件树和代码，但没有任何质量信息——代码行数、测试覆盖率、lint 通过率等。  
**影响**: 用户无法判断"这些代码质量怎么样"。  
**建议**: 在文件树顶部添加汇总指标（总文件数、总代码行、最后一次 build 状态）

### P3-30: 系统监控定位错位
**现象**: SystemMonitor 组件放在 OverviewPage 最顶部，展示 CPU / GPU / 内存 / 进程内存。这对了解 Agent 工作状态无直接帮助。  
**影响**: 占用了 OverviewPage 最宝贵的首屏位置。  
**建议**: 移到设置页或独立的"系统"面板，OverviewPage 首屏应该是项目进度

## 六、交互设计细节 — 7 个问题

### P1-31: 元 Agent 在两个地方出现，行为不一致
**现象**: 
- WishPage 右侧：内嵌的 `MetaAgentChat` 组件
- 全局右侧：可收缩的 `MetaAgentPanel` 组件

两者使用相同的后端 API 和 Zustand store 消息存储，但 UI 完全独立。当 WishPage 的对话进行到一半，用户切到 OverviewPage，右侧面板的对话历史是同步的，但 UI 状态（发送中/placeholder）不同步。  
**影响**: 用户困惑——"我应该在哪里和 AI 管家说话？"  
**建议**: 
- 统一为全局右侧面板作为唯一的元 Agent 交互入口
- WishPage 右侧改为"需求草稿编辑器"或"需求分析结果展示"

### P2-32: 工作流预设编辑器缺乏拖拽
**现象**: WorkflowPage 的编辑器 Modal 支持上移/下移按钮调整阶段顺序，但不支持拖拽排序。  
**影响**: 多阶段调整时操作效率低。  
**建议**: 添加拖拽排序支持（用 HTML5 Drag API 或 react-dnd）

### P2-33: 架构图交互反馈不足
**现象**: OverviewPage 的三层 DAG 架构图支持滚轮缩放、拖拽平移、双击下钻，但：
- 没有 hover 时的节点高亮效果（只有边高亮）
- 双击下钻没有过渡动画
- 面包屑导航位置不明显
- 底部提示文字"滚轮缩放 · 拖拽平移 · 双击下钻"非常不起眼

**影响**: 用户可能不知道可以交互。  
**建议**: 首次展示时播放一个简短的交互引导动画

### P2-34: 设置页缺乏分段保存
**现象**: SettingsPage 包含 LLM 配置、MCP 服务器配置、模型自定义定价三大块。所有设置共享一个"保存"逻辑。  
**影响**: 用户改了一个 LLM Key 想保存，但不确定是否会影响其他设置。  
**建议**: 每个 section 独立保存，或在保存时明确提示"将保存以下变更：..."

### P2-35: 团队页成员添加流程过于技术化
**现象**: TeamPage 的"配置"Tab 允许添加团队成员，需要填写角色、名称、System Prompt。  
**影响**: "System Prompt"对非程序员是完全陌生的概念。  
**建议**: 
- 提供角色模板（一键添加"多一个开发者"、"添加安全专家"）
- System Prompt 改为"角色描述"并提供预填模板

### P3-36: 验收面板 (AcceptancePanel) 缺乏详情查看能力
**现象**: 验收面板展示 Feature 列表和 PM verdict，但不能直接点击查看代码或跳转到具体文件。  
**影响**: 用户做验收决策时缺乏足够信息。  
**建议**: Feature 卡片支持展开查看关键代码变更摘要

### P3-37: 快捷键系统缺失
**现象**: 整个应用没有任何快捷键支持。  
**影响**: 高级用户无法快速导航（Ctrl+1~9 切 Tab、Ctrl+N 新项目、Space 暂停/继续等）。  
**建议**: 逐步添加核心快捷键，并在教程中说明

## 七、技术债与架构层面的体验影响 — 5 个问题

### P1-38: 前端状态管理混用 Polling + Push
**现象**: 已有事件推送基础设施（`window.automater.on('agent:log', ...)`），但大量页面仍使用 setInterval 轮询 IPC 获取数据。  
**影响**: 
- 状态更新不实时（5 秒延迟）
- 主进程负载高（频繁的 SQLite 查询）
- 数据不一致窗口（IPC 事件推送的状态 vs 轮询获取的状态可能不一致）

**建议**: 统一迁移到事件驱动架构——后端状态变更时推送到前端，前端仅在首次加载时拉取全量

### P1-39: 单文件膨胀严重
**现象**: 
- `OverviewPage.tsx`: 1459 行
- `app-store.ts`: 300+ 行状态定义 + actions
- `orchestrator.ts`: 800+ 行（拆分后仍然很大）

**影响**: 维护困难，容易引入 bug。OverviewPage 的 10+ 子组件全部内联定义在一个文件中，违反组件化原则。  
**建议**: 将 OverviewPage 拆分为独立的子组件文件（PipelineBar、InteractiveGraph、StatCard 等）

### P2-40: 大量 `any` 类型和 `try {} catch {}` 空 catch
**现象**: 前端代码中广泛使用 `any` 类型（如 `stats: any`、`project: any`），以及空 catch 块（`catch {}`、`catch (err: any)`）。  
**影响**: 
- 类型安全缺失导致运行时错误
- 空 catch 吞掉错误，调试困难

**建议**: 逐步添加类型定义，catch 块至少 console.error

### P2-41: Markdown 渲染器自实现存在局限
**现象**: DocsPage 内嵌了一个自实现的轻量 Markdown→HTML 渲染器（不使用第三方库）。  
**影响**: 复杂的 Markdown 内容（表格、脚注、任务列表等）可能渲染异常。  
**建议**: 考虑引入 `marked` 或 `react-markdown`（体积可控，功能完整）

### P3-42: Electron 打包体积 ~355MB 偏大
**现象**: CLAUDE.md 记录打包体积约 355MB。  
**影响**: 下载和安装耗时长，用户首次体验的第一道门槛。  
**建议**: 
- 检查是否打包了不必要的依赖（playwright-core 特别大）
- 考虑按需加载 playwright（用户首次使用浏览器工具时再安装）

> **总结**: AutoMater 的引擎能力和功能深度已经远超个人项目的平均水平，但产品体验层面需要一次系统性的"去技术化"改造——让不懂代码的用户也能顺畅使用。**当前最大的风险不是缺功能，而是现有功能从未在真实运行环境中验证过。** 建议下一步的第一优先级是运行时验证，然后才是体验优化。
